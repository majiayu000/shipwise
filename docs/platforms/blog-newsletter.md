# Blog And Newsletter Source Guide

Blog posts and newsletters are durable explanation channels. They are best when the project has a story, method, benchmark, architecture, or lessons learned.

## Sources To Check

Each publication has its own guidelines. Before submitting, check the target publication's current rules:

- Personal blog: your own editorial rules.
- DEV Community: https://dev.to/code-of-conduct
- Hashnode: check publication/community rules.
- Newsletter: check submission or sponsor guidelines.

## Use When

- The project needs more explanation than a social post can hold.
- There is a benchmark or architecture story.
- You want long-tail SEO.
- You need a source article that other posts can link to.

## Agent Checklist

- [ ] The post has a concrete thesis.
- [ ] It includes setup, method, result, and caveats.
- [ ] Code snippets are tested.
- [ ] Claims are sourced.
- [ ] CTA is specific: try, star, comment, file issue, or subscribe.

## Do Not

- Do not write a generic launch essay with no proof.
- Do not syndicate the same post everywhere without canonical link handling.
- Do not hide sponsorship or affiliation.
